# 功能文件生成指令
> 说明：LLM 读取本文件 + 需求文档 + 架构决策文档后，自动生成各模块的功能文件。人只负责裁剪与确认。

---

## 你的任务

根据提供的需求文档和架构决策文档，为每个模块生成对应的功能文件。

输出格式严格遵循 `feature_spec.md` 中的结构，不得自行发明字段。
每个模块输出一份独立的功能文件，文件名格式：`feature_[模块名].md`

---

## 生成规则

### 第一步：读取架构模块清单

从架构决策文档中获取模块清单，每个模块对应一份功能文件。

如果没有架构决策文档：
- 先从需求文档中识别功能边界，提出模块划分建议
- 标记为 [PENDING]，等人确认模块划分后再生成功能文件

### 第二步：识别每个模块的边界

对每个模块，从需求文档中提取：

**包含什么：**
- 需求文档中明确属于此模块的功能点
- 只取明确描述的，不做推断扩展

**不包含什么：**
- 需求文档中明确排除的
- 逻辑上容易被误认为属于此模块、但实际属于其他模块的
- 没有被提及的——默认不包含，不做假设

### 第三步：梳理技术边界

**依赖关系：**
- 此模块需要调用哪些其他模块或外部服务
- 从需求文档的业务流程中推断，不确定的标记 [PENDING]

**被依赖关系：**
- 哪些模块会调用此模块
- 同上，不确定的标记 [PENDING]

**不可触碰：**
- 根据总纲的技术禁区和行为禁区推导
- 涉及高风险操作（支付、删除、权限）的，明确写出不可直接访问的内容

### 第四步：定义接口契约

从需求文档的业务流程中推导对外接口：
- 只定义此模块需要暴露给其他模块或外部的接口
- 入参出参从业务需求推导，不确定的字段标记 [PENDING]
- 错误码按照业务语义定义，不随意发明

### 第五步：评定风险等级

使用风险矩阵自动评定：

后果严重度（勾选适用项）：
- 涉及金钱、支付、计费
- 数据删除或不可逆修改
- 权限、认证、访问控制
- 影响所有用户的全局配置
- 对外部系统产生副作用
- 合规/法律相关

出错概率（勾选适用项）：
- 复杂条件分支
- 并发/竞态条件
- 外部服务依赖
- 需求描述含 [PENDING]
- 新引入的技术、库、框架
- 跨模块数据流转

定级规则：
- 后果清单任意勾选 → 至少 L2
- 合计 0-1 项 → L1，2-3 项 → L2，4项以上 → L3

### 第六步：判断是否需要UI规范

- 此模块有用户界面 → 在功能文件中填写 UI 说明节，引用 ui_spec.md
- 需求文档中有设计稿或样式描述 → 提炼特殊 UI 要求
- 完全没有 UI 相关描述 → 跳过此节

### 第七步：标记不确定项

遇到以下情况，标记为 [PENDING]，不要自行决定：
- 需求文档描述模糊或有歧义
- 模块边界不清晰，可能属于多个模块
- 接口字段无法从需求文档推导
- 业务规则有多种可能的理解方式
- 依赖关系不确定

---

## 输出要求

1. 每个模块输出一份独立文件，格式严格遵循 `feature_spec.md`
2. 所有 [PENDING] 项必须说明为什么不确定，方便人决策
3. 生成完成后，输出一段生成摘要：
   - 共生成几个功能文件
   - 各模块风险等级分布
   - 有几个 [PENDING] 项需要人确认
   - 模块间依赖关系图（文字版）
   - 建议人重点审查哪些模块

---

## 人的裁剪职责

LLM 生成后，人只需要做四件事：
1. **确认**模块划分是否合理，有无遗漏或重复
2. **确认**所有 [PENDING] 项，补充缺失信息
3. **调整**模块边界（如有跨模块职责不清晰的地方）
4. **审查**高风险模块（L3）的接口契约是否准确

不需要从头写功能文件。

# WBS 生成指令
> 说明：LLM 读取本文件 + 功能文件后，自动生成 WBS 任务单。人只负责裁剪与确认。

---

## 你的任务

根据提供的功能文件，生成该模块的 WBS 任务单。

输出格式严格遵循 `wbs_format.md` 中的结构，不得自行发明字段。

---

## 生成规则

### 第一步：判断功能类型

阅读功能文件，判断每个功能点属于哪种类型：
- **全栈**：有数据模型 + 业务逻辑 + 接口 + 前端
- **纯后端**：只有数据模型 + 业务逻辑 + 接口，跳过步骤 4、5
- **纯前端**：以组件和接入为主，步骤 1、2 可能很薄或跳过

### 第二步：拆解任务

按垂直切片拆解，每个任务必须满足最小单元原则：
- 有且只有一个完成标准
- 可以被独立测试，不依赖同批其他任务
- 失败时能明确定位问题

**拆解粒度判断：**
- 如果一个任务有多个完成标准 → 继续拆分
- 如果两个任务的完成标准必须同时验证 → 合并为一个
- 如果不确定是否拆分 → 拆分，人裁剪比人补充容易

### 第三步：定义完成标准

每个步骤的完成标准必须是可验证的语言：
- ❌ "功能正常" / "测试通过" / "实现完成"
- ✅ "POST /auth/login 正确凭证返回 200 + token，错误凭证返回 401"
- ✅ "login(email, password) 正确凭证返回 user 对象，错误凭证抛出 AuthError"
- ✅ "migration 跑通，users 表包含 id/email/password_hash 字段"

### 第四步：评定风险等级

对照功能文件中的风险等级，每个任务的风险等级不得高于模块风险等级。

使用风险矩阵自动计算：
- 后果清单勾选任意一项 → 至少 L2
- 合计 0-1 项 → L1，2-3 项 → L2，4项以上 → L3

### 第五步：设置检查点

根据风险等级自动设置检查点位置：
- L1 模块：所有任务完成后一个检查点
- L2 模块：每 3-5 个任务一个检查点
- L3 模块：每个任务完成后一个检查点

### 第六步：标记不确定项

遇到以下情况，标记为 [PENDING]，不要自行决定：
- 功能文件中有模糊描述
- 需要依赖尚未确认的外部接口
- 涉及业务规则但功能文件未明确说明
- 技术方案有多种可能，无法从约束中判断

---

## 输出要求

1. 严格按照 `wbs_format.md` 格式输出
2. Task 编号从 001 开始顺序递增
3. 每个任务的输入条件必须引用具体的 Task 编号，不写模糊描述
4. 所有 [PENDING] 项集中列在任务末尾，并说明为什么不确定
5. 生成完成后，输出一段简短的生成摘要：
   - 共生成几个任务
   - 各风险等级分布
   - 有几个 [PENDING] 项需要人确认
   - 建议人重点关注哪些任务

---

## 人的裁剪职责

LLM 生成后，人只需要做三件事：
1. **合并**粒度过细的任务（两个任务加在一起仍然可独立验证）
2. **确认**所有 [PENDING] 项，补充缺失信息
3. **调整**执行顺序（如有依赖关系判断有误）

不需要逐行审查代码级细节。
